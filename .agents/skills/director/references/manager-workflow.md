# Manager Workflow Reference

Read this file at Step 3 (Compose Prompts) when triage selects manager mode. It stays in context through Steps 4-9.

## Prompt Templates

### Required in Every Manager Prompt

- The task with success criteria
- "Read brain/index.md first for project conventions."
- "Use the worktree skill. Work in worktrees, never on main."
- Relevant context from brain files — known gotchas, constraints, domain quirks
- The manager's SKILL.md contains the worker contract. Include any task-specific gotchas in the manager prompt.
- **Explicit negative file list** when parallel managers share a codebase: "Do NOT modify or delete: <files owned by other managers>". Codex workers interpret "clean up" broadly and will delete files they consider unused.

> **CRITICAL: Use the `worktree` skill for ALL worktree operations — create, exec, merge, cleanup. Never run raw `git worktree add/remove` directly. The skill enforces CWD safety checks that prevent shell death. If your shell dies (every command returns exit code 1 with no output), stop — the director will restart you.**

### Conditional Inclusions

**When editing `.claude/` files:**

> **Gotcha: `.claude/skills/` is a symlink to `.agents/skills/`.** Git tracks files under `.agents/`, so `git diff`, `git status`, and `git add` show `.agents/` paths. Use `.agents/` paths for git operations. Also: concurrent sessions may leave dirty files on main — `git stash` before merging your worktree branch, then `git stash pop` after.

**When multiple managers touch overlapping files:**

> **Merges are serialized by lockfile.** The worktree CLI acquires `.worktrees/.merge-lock` before merging and releases it after. Concurrent merge attempts fail with a clear error. No special instructions needed in manager prompts.

### Context Manifest

Manifests are auto-generated by `noodle spawn`. No separate Write call needed. The manifest records what context was in the prompt file and enables post-hoc debugging of "why didn't the manager know about X?"

To check a manifest after the fact: `cat ~/.noodle/projects/<hash>/sessions/<id>/mgr-<name>.manifest.json`

### Verification Commands

Front-load the expected workflow: implement -> lint/format -> test -> commit -> merge. Include exact commands:

- Go: `go -C <worktree>/old_noodle test ./... && go -C <worktree>/old_noodle vet ./...`
- Go: `cd <worktree>/<module> && go build ./... && go vet ./... && go test ./...`
- Explicitly state: "Run lint and fix ALL issues before committing. Do not leave lint violations for cleanup."

When the codebase has strict linting (oxlint, clippy -D warnings), mention specific lint rules that commonly trip up generated code (eqeqeq, prefer-destructuring, explicit-function-return-type for oxlint; unused imports for clippy).

### Review Manager Prompt

Include all of the above, plus:

- "Use the `review` skill (invoke via `Skill` tool) to review the changes."
- "You are running autonomously — do not use `AskUserQuestion`. Determine scope from context (use BIG CHANGE mode) and produce a complete review without pausing between sections."
- The original objective and success criteria
- **If plan-based**: the plan file path and "Read the plan and verify every requirement and acceptance criterion."
- Which worktrees/branches the managers worked in
- Instruction to produce a verdict: **pass** or a list of specific issues with actionable fixes
- "Tests were already verified by the manager. Focus on code quality and correctness, not test execution."

## Commands Reference

**CRITICAL: Never use variable assignments in bash commands** — they trigger permission prompts. Inline literal paths directly. See `brain/codebase/claude-subprocess-spawn-patterns.md`.

All commands use the pre-built binary at the project root: `./noodle`

### Session Lifecycle

```bash
# Initialize a new session (prints session ID to stdout)
./noodle init --project "$(pwd)"

# Clean up all session resources when done
./noodle cleanup --session <id>
```

### Spawning

```bash
# Spawn a standard manager (delegates to workers)
./noodle spawn --session <id> --name <name> --prompt-file <path>

# Spawn a direct manager (does all work itself, no delegation tools)
./noodle spawn --session <id> --name <name> --prompt-file <path> --agent operator

# Spawn with dep checking in a worktree
./noodle spawn --session <id> --name <name> --prompt-file <path> --worktree <path>

# Resume a stopped manager
./noodle resume --session <id> --agent <agent-id>
```

`spawn` handles all ceremony: prompt validation, template injection, manifest generation, dep checking, tree registration, orphan checking, and tmux launch with the stamp pipeline.

### Monitoring (primary — use these)

```bash
# Block until all managers finish
./noodle wait --session <id> --until all_done --timeout 30m

# Block until a specific manager exits (for incremental review spawning)
./noodle wait --session <id> --until agent_exited --agent <id> --timeout 15m

# Block until any failure (for proactive intervention)
./noodle wait --session <id> --until failure_detected --timeout 5m

# Compact status (context-safe: 1 line per agent + summary)
./noodle status --session <id>

# Full agent hierarchy
./noodle tree --session <id>

# Detailed tree with file changes
./noodle tree --session <id> --detail
```

### Query (use during review and performance analysis)

```bash
# Cost rollup from agent tree (instant — pre-computed)
./noodle costs --session <id>

# Worker status for a specific manager
./noodle worker-status --session <id> --manager <manager-name>

# Session overview
./noodle sessions --session <id>

# Failures
./noodle failures --session <id>
```

### Diagnostics (use only when investigating a specific problem)

These commands produce verbose output. **Do not run them in a loop.** Use them once to diagnose a failure, then return to `status`/`wait`.

```bash
# Investigate a specific manager's recent output
./noodle logs --session <id> --agent <name> [--lines 20]

# Event timeline (post-mortem or debugging)
./noodle events --session <id> [-n 20] [--json]

# Get the Claude session ID for a manager (needed for resume)
./noodle session-id --session <id> --agent <name>
```

### Recovery

```bash
# List recoverable agents with attempt counts
./noodle recover --session <id> --list

# Preview recovery context
./noodle recover --session <id> --agent <agent-id> --dry-run

# Execute recovery (enforced: retry limit, kill+resume, auto-context)
./noodle recover --session <id> --agent <agent-id>

# Manual kill (use recover instead when possible)
./noodle kill --session <id> --agent <agent-id>

# Kill all managers in a session
./noodle kill --session <id> --all
```

### Coordination

```bash
# Monitor manager health and blockers
./noodle status --session <id>
./noodle worker-status --session <id> --manager <manager-name>

# Emergency stop
./noodle kill --session <id> --all
```

### Summarize

```bash
# Generate session summary after completion
./noodle summarize --session <id> --objective "<request>" [--learnings "<notes>"] [--json]
```

## Performance Review

Run `./noodle costs --session <id>` and review the tree-based cost rollup. See [references/soul.md](references/soul.md) for principles.

### Analysis Checklist

1. **Total cost** — report to the user as part of the completion summary
2. **Per-manager breakdown** — identify which managers were expensive and why
3. **Wasted spend** — managers killed and restarted (double cost), workers that failed and retried, unnecessary decomposition (single-task work split into multiple workers)
4. **Process issues** — wasted turns, prompt gaps, decomposition mistakes, tool misuse
5. **Dashboard gaps** — missing signals, false positives, friction. Add improvement todos to `brain/todos.md`.

### Efficiency Questions

- Could fewer managers have done the work?
- Would direct mode have been cheaper for small/verification tasks?
- Were worker prompts specific enough, or did vague prompts cause wasted turns?
- Would a different decomposition have reduced overlap or merge conflicts?

Include 2-3 **specific efficiency suggestions** in the completion summary.

### Route Learnings

Extract high-level principles, not implementation notes:
- Delegation principles -> `brain/delegation/`
- Codebase gotchas -> `brain/codebase/`
- Process improvements -> `brain/delegation/`
